### 2.2 模块关系分析

**良好设计**:
- `src/config.py` - 统一配置管理，使用dataclass+环境变量，支持多环境
- `src/cache/` - 多级缓存(内存LRU+Redis)，带穿透保护和雪崩防护

- `src/db/` - 异步数据库封装(asyncpg+SQLAlchemy)

**设计缺陷**:
- Flask + FastAPI 双框架并存，backup目录包含旧Flask代码，未完全清理
- `src/__init__.py` 暴露了过多内部模块，增加耦合风险
- `src/fetcher/__init__.py` 导入链复杂，存在潜在的循环依赖

## 4. 详细问题描述

### P0-1: funds表Schema不一致

**位置**: `db/init.sql` vs `db/pool.py`  
**描述**: `db/init.sql`第34行定义`funds`表主键列为`code VARCHAR(10) UNIQUE NOT NULL`，而`db/pool.py`第101行定义相同表为`fund_code VARCHAR(20) UNIQUE NOT NULL`。Docker Compose使用init.sql初始化时创建的是`code`列，但`db/fund_ops.py`所有操作都使用`fund_code`列名。

**风险**: 
- 全新部署(PostgreSQL容器首次启动)时，`funds`表使用init.sql schema
- `fund_ops.get_fund_info()`执行`SELECT * FROM funds WHERE fund_code = %s`会报列名不存在错误
- 现有数据无法写入，`fetch_fund_detail_enhanced()`等所有依赖数据库的增强功能全部失败

**修复方案**:
```python
# 方案A: 统一init.sql使用fund_code列名
# init.sql第34行改为:
code VARCHAR(20) UNIQUE NOT NULL  # 并改为fund_code

# 方案B: 统一pool.py使用code列名  
# pool.py第101行改为:
code VARCHAR(10) UNIQUE NOT NULL  # 与init.sql一致
```
**建议**: 采用方案A，20字符长度更合理(部分基金代码含字母后缀如`000001.OF`)

### P0-6: Redis不可用时安全函数绕过

**位置**: `src/cache/redis_cache.py`  
**问题函数**:
```python
def is_account_locked(username: str) -> bool:
    client = get_redis_client()
    if client is None:
        return False  # 缺陷: 静默降级为未锁定
```

**攻击场景**: 
1. Redis服务宕机或网络故障
2. 攻击者使用任意密码尝试登录(无限制暴力破解)
3. `is_account_locked()`返回`False`，`increment_login_fail_count()`也因Redis不可用返回0
4. 账户永远不会真正锁定

**修复**:
```python
def is_account_locked(username: str) -> bool:
    client = get_redis_client()
    if client is None:
        logger.error("Redis不可用，无法判断账户锁定状态，安全降级为锁定")
        return True  # 安全降级: Redis不可用时视为锁定
```

### P1-10: PBKDF2迭代次数偏低

### P1-12: startPeriodicFetch无登录检查

**位置**: `web/vue3/src/stores/fund.ts` 第61行

```typescript
startPeriodicFetch(): void {
    setInterval(() => { this.fetchFunds(true) }, 10 * 60 * 1000)
    // 用户登出后仍继续执行...
}
```

**修复**:
```typescript
startPeriodicFetch(): void {

    this._periodicTimer = setInterval(() => {
        if (this.user) {  // 仅在已登录时请求
            this.fetchFunds(true)
        }
    }, 10 * 60 * 1000)
},
stopPeriodicFetch(): void {
    if (this._periodicTimer) {
        clearInterval(this._periodicTimer)
    }
}
// 在logout()中调用stopPeriodicFetch()
```

## 5. 优秀实践

### 5.1 配置管理 (`src/config.py`)
- 使用dataclass定义配置类型，IDE自动补全
- `from_env()`模式统一，环境变量与代码解耦
- 生产环境验证失败直接抛异常，开发环境仅警告
- 提供`to_dict()`用于调试不泄露敏感信息

### 5.2 多级缓存架构 (`src/cache/`)
- LRU内存缓存(快速) → Redis缓存(共享) → 数据源(慢速)
- 缓存穿透保护: 空值标记`__NULL__`短时缓存
- 缓存雪崩防护: TTL添加随机抖动(±30秒)
- 统计指标完善: hits/misses/evictions/hit_rate

### 5.3 评分系统设计 (`src/scoring/`)
- 8维度独立模块，职责单一
- 权重可配置验证，总和必须=100
- 支持评分报告格式化输出
- 排名加分机制(`apply_ranking_bonus`)拉开分数差距

### 5.4 异步数据库封装 (`src/db/async_database.py`)
- 单例+异步连接池封装
- 失败重试机制(可配置重试次数和延迟)
- 上下文管理器支持`async with`
- 连接池预热`warmup()`方法

### 5.5 分布式调度器 (`src/scheduler/manager.py`)
- APScheduler+Redis JobStore持久化
- Redis分布式锁保证集群单实例执行
- 事件监听器跟踪job执行统计
- 飞书/钉钉告警webhook集成

### 5.6 前端状态管理 (`web/vue3/src/stores/fund.ts`)
- Pinia store，TypeScript类型安全
- localStorage缓存降级方案
- 并行`Promise.all`批量加载
- 统一错误格式化和用户提示

### 5.7 安全机制
- PBKDF2密码哈希(虽然迭代次数偏低)
- JWT token黑名单(Redis存储)
- 账户锁定(5次失败后15分钟锁定)
